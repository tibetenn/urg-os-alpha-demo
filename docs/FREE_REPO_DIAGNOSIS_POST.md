# あなたの OSS リポジトリを URG-OS で診断してみませんか

> **これは企業向け営業文書ではありません。**  
> 公開 OSS リポジトリのメンテナ・個人開発者を対象にした、検証参加の案内です。

---

## URG-OS とは (一行で)

グラフ構造でコードの「いま」を可視化する Code Governance AI です。コードを書き換えません。

---

## 何をするのか

あなたの公開 GitHub リポジトリ (Python) を対象に、URG-OS pipeline を実行します。

pipeline は以下を行います。

1. リポジトリを一時的に clone
2. AST 解析でコードをグラフ構造に変換
3. モジュール・関数・クラス・import・呼び出し関係を抽出
4. セキュリティや実行時に関連しうる静的構造の候補を整理
5. パターンを集約・匿名化し、Code Governance Report を生成
6. **clone したファイルを削除**

あなたのコードそのものは保存されません。レポートに残るのは、ノード型・エッジ型・件数・集約統計などの構造情報です。

---

## レポートに含まれるもの

| セクション | 内容 |
|-----------|------|
| Graph Overview | ノード数・エッジ数・主要なノード種別 / エッジ種別 |
| Code Structure Summary | モジュール・関数・クラス間の依存関係、呼び出し構造の集約 |
| Code Quality Candidates | 複雑になりやすい構造、到達しにくい可能性のあるパス、依存の偏りなどの候補 |
| Security-oriented Structure | 外部入力・認証境界・シークレット参照など、注意して確認したい構造候補 |
| Runtime-oriented Structure | 設定依存・グローバル状態・副作用など、実行時に影響しうる静的構造 |
| Privacy Scan | raw コードや識別情報がレポートに混入していないかの確認 |

URG-OS は、現時点では「問題を断定する診断ツール」ではなく、コード構造を観察し、レビューの出発点になる情報をまとめる Alpha Demo です。

---

## レポートに含まれないもの

- コードの断片や行
- ファイル名・関数名・クラス名 (raw コード由来のもの)
- リポジトリの URL やオーナー名
- シークレットや API キー
- 「このコードはここが悪い」という断定的評価

URG-OS は**観察・記述**するツールです。良い/悪いを評価するのはあなた自身です。

---

## 対象リポジトリの条件

- GitHub で公開されている Python リポジトリ
- Python 3.6 以上を主な対象言語としているもの
- ライブラリ・ツール・CLI・Web アプリ・スクリプト集、問いません
- スター数・コミット数・規模は問いません

---

## なぜ無料で提供するのか

URG-OS はまだ検証フェーズです。1,000 件の公開 OSS を処理した実績はありますが、それは**自動収集したリポジトリ群**です。

メンテナ本人が「診断してみたい」と思っているリポジトリに対して pipeline を実行することで、レポートの有用性・説明のわかりやすさ・実際の開発者にとって役立つ観点を改善したいと考えています。

フィードバックがあれば歓迎します。

- 一言でどんなプロジェクトか
- 診断結果で特に知りたいこと (あれば)

連絡先: GitHub Issues または README に記載のコンタクト先
