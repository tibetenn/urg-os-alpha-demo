> **注記**: 以下の統計はすべて公開 OSS リポジトリの集約・匿名データです。  
> リポジトリ名・オーナー名・コード断片は含まれていません。

## 失敗の内訳

失敗した 71 件は主に技術的なアクセス問題であり、pipeline 全体は停止せずに処理を継続しました。

| 原因 | 件数 | 説明 |
|------|------|------|
| `auth_required` | 55 | パブリックに見えて認証が必要なリポジトリ |
| `empty_repo` | 11 | Python ファイルが存在しないためスキップ |
| `clone_failed` | 5 | 削除済み / 移動済み / ネットワークエラー |

## Graph 特性 (集約統計)

以下は 929 件の成功リポジトリから得た、匿名化されたグラフ構造の集約統計です。個別リポジトリは特定できません。

### ノード・エッジ密度の傾向

成功リポジトリ全体で見られた、おおまかな構造傾向です。

- **小規模 repo (< 10 ファイル)**: グラフが比較的疎で、依存エッジが少ない傾向があります。単機能ライブラリに多く見られます。
- **中規模 repo (10–100 ファイル)**: モジュール間依存や呼び出し構造が複雑化し始めます。
- **大規模 repo (100+ ファイル)**: 呼び出し関係・import・class / method の分布が広がり、レビュー時に構造の把握が重要になります。

### Code Governance Report で表現しうる構造パターン例

現時点の Alpha Demo では、URG-OS は公開リポジトリの静的コード構造をグラフ化し、ノード種別・エッジ種別・依存関係・呼び出し構造の集約統計を出力します。

以下は、今後の Code Governance Report でより詳しく分類していく予定の構造パターン例です。  
特定のリポジトリに対する断定的な脆弱性診断ではありません。

| 分類 | 構造パターン例 |
|---------|---------|
| Security-oriented Structure | 外部入力から重要処理へつながる呼び出し構造、認証境界が複雑になりやすい構造、シークレット参照が分散しやすい構造 |
| Runtime-oriented Structure | 設定値が実行経路に影響しうる構造、グローバル状態の変更が起きやすい構造、副作用を持つ初期化処理 |
| Code Quality Candidates | 依存が集中しているモジュール、呼び出し関係が深い構造、到達しにくい可能性のある関数群 |

この段階では、URG-OS は「問題を断定する」のではなく、レビューや改善検討の入口になる構造情報を提示します。

## Privacy Safety Scan 結果

Safety Scan は 4 ステージすべてで自動実行されました。

| チェック項目 | 1000 repo 結果 |
|------------|---------------|
| raw コード | ✅ 含まれない |
| ファイルパス | ✅ 含まれない |
| リポジトリ URL | ✅ 含まれない |
| オーナー / リポジトリ名 | ✅ 含まれない |
| シークレット名 | ✅ 含まれない |
| メールアドレス | ✅ 含まれない |
| 実コード由来の識別子 | ✅ 含まれない |

**False Positive**: 2 件 (全 4 ステージを通じて同一の 2 件)  
PatternStore の description 内の英単語 (`token` 等) が正規表現に一致したもの。`isalpha()` + suffix チェックで FP と判定済みです。実際のシークレットリークではありません。

メモリは本番環境では streaming / per-batch flush により削減可能です。現時点では機能検証フェーズとして許容範囲内です。

## このレポートが意味すること

1,000 件の公開 OSS リポジトリを処理しても、raw コードや識別情報はレポートに残りません。  
グラフ構造の**集約パターン**のみが残り、それを元に Code Governance Report が生成されます。

これが URG-OS の設計原則「**観察するが、コードそのものは保存しない**」の実証です。
